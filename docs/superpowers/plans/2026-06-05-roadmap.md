# Feuille de route d'implémentation — AI Subtitle Pro (Go/Wails)

> Découpage en 5 plans par **jalons**. Chaque jalon laisse un logiciel qui se lance et se teste.
> La spec de référence : `docs/superpowers/specs/2026-06-05-go-subtitle-translator-design.md`.

| Plan | Jalon | Contenu | « Done » = |
|---|---|---|---|
| **1. Fondations** | App vide qui tourne | module Go, scaffold Wails+Svelte, paquets `config` (TDD) + `runtime` (chemins), fenêtre sombre | `wails build` produit un .exe qui s'ouvre ; `go test ./...` vert |
| **2. Sous-titres & MKV** | Extraire/remuxer en CLI | paquet `subs` (parse SRT/ASS, découpage lots+contexte, réassemblage — TDD) + paquet `mkv` (scan `-J`, extract, mux) | un test/CLI extrait une piste, la réécrit, la remuxe |
| **3. Moteurs** | Traduire un lot | interface `Translator` + `gemini` (TDD parsing JSON) + `local` (cycle de vie `llama-server`) | chaque moteur traduit un lot de lignes |
| **4. Pipeline & intégration** | Backend complet | paquet `app` (orchestration, events progress/log/done, annulation `context`, mode Test 20s) + embarquage réel des binaires (`go:embed`) dans `runtime` | pipeline complet déclenchable côté Go |
| **5. Interface** | App utilisable | UI Svelte (drop, sélecteurs moteur/langue/piste, progression animée, journal), branchée aux bindings/events, style sombre épuré | traduction de bout en bout depuis l'UI |

**Pourquoi détailler les plans 2 à 5 « juste à temps » :** leur code dépend du scaffold et des signatures réellement produits au plan précédent. Les détailler à l'avance produirait du code spéculatif (anti-DRY/YAGNI). Chaque plan sera rédigé en entier juste avant son exécution.

**Principes communs :** TDD sur la logique pure, commits fréquents et petits, paquets à responsabilité unique, sous-processus lancés sans fenêtre console (`CREATE_NO_WINDOW`), aucun secret ni gros binaire versionné.
